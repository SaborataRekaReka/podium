# Аудит проекта Podium (быстрый engineering review)

## Что уже хорошо

- Проект собран как полностью статичный лендинг без сборщика: быстрый старт и деплой (`index.html` + `styles.css` + `script.js`).
- Есть базовые меры доступности: `lang="ru"`, `skip-link`, семантические секции, `aria-label` в интерактивах, клавиатурная навигация в табах.
- Используется lazy-load изображений через `data-src` + `IntersectionObserver`.
- Учтён `prefers-reduced-motion` и в JS, и в CSS.


## Что уже внедрено (текущий шаг)

- Добавлена базовая automation-проверка без внешних зависимостей:
  - `npm run check:js` — синтаксис JS,
  - `npm run check:links` — проверка локальных `src/href` в `index.html`.
- Улучшена UX-связка кнопки «Пройти квиз»: после прокрутки добавлен фокус на контент секции.
- Для горизонтальных каруселей вынесена общая клавиатурная навигация (ArrowLeft/ArrowRight) в единый helper, чтобы поведение было консистентным между блоками.

## Ключевые риски

1. **Монолитный JS-файл (~200К+)**
   - Большой объём контента и логики в одном `script.js` затрудняет поддержку, ревью и локализацию.
2. **Монолитный CSS-файл (~200К+)**
   - В `styles.css` видны «слои» разных итераций (например, повторные `:root` и наборы hero-стилей), что повышает риск регрессий и переопределений.
3. **Смешение данных и представления**
   - Длинные контентные блоки и HTML-шаблоны находятся в JS через `innerHTML`, что усложняет редактуру контента и тестирование.
4. **Отсутствуют автоматические проверки**
   - Нет package-скриптов и CI-валидации (линт HTML/CSS/JS, проверка ссылок, базовая accessibility-проверка).

## Рекомендации по архитектуре

### 1) Разделить `script.js` на модули

Предлагаемая структура:

- `src/data/content.js` — только тексты/метаданные.
- `src/ui/tabs.js`, `src/ui/carousel.js`, `src/ui/quiz.js`, `src/ui/tooltips.js` — поведение компонентов.
- `src/utils/dom.js`, `src/utils/a11y.js`, `src/utils/media.js` — утилиты.
- `src/main.js` — точка инициализации.

Минимальный шаг без сборщика: ES-модули нативно (`<script type="module">`) и несколько файлов.

### 2) Вынести контент в JSON/MD

- Исторические тексты, карточки и описания шин хранить в `assets/data/*.json`.
- JS оставить «тонким»: загрузка данных и рендер.
- Это ускорит обновления маркетинговых блоков без правок сложной логики.

### 3) Сократить прямой `innerHTML`

- Для пользовательских/внешних данных — `textContent` и шаблонизатор на `createElement`.
- Для статичных шаблонов — `<template>` в HTML + клонирование.

## UX-улучшения (приоритет)

### Высокий приоритет

1. **Привязать кнопку «Пройти квиз» к фактическому сценарию**
   - Сейчас кнопка в hero визуально заметная; важно добавить скролл/открытие квиза по клику + фокус-менеджмент.
2. **Навигационный прогресс по сторителлингу**
   - Добавить индикатор текущего раздела/прогресса чтения (особенно при длинном скролле).
3. **Ускорить perceived performance для hero**
   - Preload LCP-изображения и заранее фиксировать размеры важных медиа, чтобы сократить CLS.

### Средний приоритет

4. **Единый паттерн каруселей и табов**
   - Сейчас несколько близких интерактивных блоков; стоит унифицировать API и поведение (кнопки, пагинация, swipe, клавиатура).
5. **Системные состояния UI**
   - Добавить консистентные состояния `loading / empty / error` для данных (например, квиз при сбое загрузки).
6. **Микро-взаимодействия**
   - Больше явной обратной связи: активные пункты, hover/focus-visible, анимации «подтверждения» после действия.

## Стили и дизайн-система

### 1) Свести токены в один источник

- Оставить единый блок design tokens (`--color-*`, `--space-*`, `--radius-*`, `--z-*`, `--motion-*`).
- Убрать дубли/переопределения между ранними и поздними блоками стилей.

### 2) Нормализовать нейминг и слои

- Зафиксировать соглашение (BEM/utility mix), слой `base/components/utilities/overrides`.
- Критические override-блоки документировать рядом с причиной.

### 3) Адаптив и типографика

- Проверить диапазоны `clamp()` и брейкпоинты на устройствах 320–1440.
- Вынести повторяющиеся паттерны карточек в переиспользуемые классы-компоненты.

## Скрипты, качество и процессы

### Быстрый минимальный набор (без тяжёлого рефакторинга)

- `npm run lint:js` (ESLint)
- `npm run lint:css` (Stylelint)
- `npm run lint:html` (html-validate)
- `npm run format` (Prettier)
- `npm run test:a11y` (pa11y/axe для 2-3 ключевых экранов)
- `npm run test:links` (проверка битых ссылок/ассетов)

### CI (GitHub Actions)

- PR pipeline: lint + smoke checks + сбор артефактов (скриншоты ключевых экранов).
- Бюджеты производительности (Lighthouse CI) по LCP/CLS/INP.
